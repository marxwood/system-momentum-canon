# Operating Model and Architecture

Every organization has an architecture.

Even when it denies it.

Architecture is not only technical.

Architecture is the shape of:

- work routing
- authority boundaries
- permission surfaces
- and repair loops

This section treats operating model as system design.

---

## Architecture is not diagrams

Architecture is not a slide deck.

Architecture is the set of constraints that determine:

- what can happen
- what cannot happen
- and what happens by default

Defaults matter more than policies.

---

## The organizational hazard

The hazard is invisible architecture.

Invisible architecture produces:

- shadow execution
- fragmented truth
- and unbounded capability

Unbounded capability becomes capture.

---

## What this section covers

This section defines:

- organization structure that supports bounded delegation
- team composition and domain boundaries
- shared execution fabric as infrastructure
- tool ecosystem discipline
- capability boundaries and least privilege
- human + execution-unit workflow integration

---

## Minimal implementation

A correction-capable organization must preserve:

1. explicit boundaries for capabilities and authority
2. shared infrastructure for execution and audit
3. structures that avoid bottleneck heroism
4. architecture changes as corrigible change

If architecture is invisible, power routes through relationships.
