# Brain

This document defines **Brain** as the **unique canonical derivative**
and the **canonical runtime model** (conceptual)
through which the System Momentum Canon is enacted.

Brain is a **canonical construct**, not a technical system.
It exists to ensure that the Canon is not merely declarative,
but executable as a normative system.

---

## 1. Canonical Status

Brain is part of the **Canon** and occupies a **derivative position** within it.

It is:

- the **only canonical derivative**
- mandatory
- unique
- non-optional
- non-variant

There is exactly **one Brain**.

All canonical systems derive from and operate through the same Brain.
Any system not conforming to Brain is non-canonical by definition.

---

## 2. Brain as Canonical Derivative

As a **canonical derivative**, Brain exists to materialize Core constraints
into a concrete system model.

Brain:

- derives from Core (Constitution and Authority)
- is constrained by all Core invariants
- introduces no new norms or authority
- provides the sole canonical model through which the Canon becomes operative

Brain does not compete with Core.
Brain cannot exist without Core.
Core cannot be enacted without Brain.

---

## 3. Role of Brain

Brain exists to prevent two systemic failures:

1. **Normative abstraction collapse**  
   (The Canon exists, but no concrete model enforces it.)

2. **Implementation-driven modeling**  
   (Concrete systems invent structures first and justify them retroactively.)

Brain defines **what must exist and occur**
for the Canon to be enacted as a system,
without defining **how it is technically implemented**
or **how it is practically used**.

---

## 4. Brain as Canonical Runtime Model (Conceptual)

Brain is the **canonical runtime model** of the Canon.

Canonical runtime model means:

- the obligatory epistemic execution of canonical rules
- the ordered phases through which claims, meaning, and validity are processed

Canonical runtime is **conceptual and normative**, not technical.

Brain does **not** define:

- processes
- event loops
- schedulers
- memory models
- tooling
- infrastructure
- deployment

---

## 5. Canonical Functional Domains

Brain defines a **closed set of Canonical Functional Domains**.

These domains represent **non-negotiable separations of epistemic responsibility** within the Brain.  
They do **not** define execution order, flows, or technical behavior.

Each domain establishes:

- what kind of epistemic activity may occur within it,
- what it is explicitly prohibited from doing,
- and where binding authority begins and ends.

The Canonical Functional Domains are:

- **Perceptual Domain**  
  Responsible for intake of signals and observations.  
  This domain does not assert truth, assign meaning, or create obligation.

- **Cognitive / Reasoning Domain**  
  Responsible for thinking, inference, and exploration.  
  This domain produces hypotheses and internal reasoning artifacts, but has no binding authority.

- **Knowledge / Semantic Authority Domain (KSA Domain)**  
  Responsible for authoritative meaning, classification, and invariants.  
  This domain is the sole authority over semantics and definitions.

- **Institutional / Epistemic Memory Domain (IEM Domain)**  
  Responsible for institutional commitment, provenance, and accountability over time.  
  This domain preserves claims, decisions, and their validity status.

- **Reflective Domain**  
  Responsible for critical evaluation of institutional state.  
  This domain produces assessment and critique without altering recorded commitments.

- **Action / Governance Domain**  
  Responsible for authoritative decision and execution.  
  This domain marks the boundary where thinking ends and obligation begins.

The set of Canonical Functional Domains is **closed**.
No domain may be added, removed, or redefined without an explicit Canon amendment.

---

## 6. Canonical Object Model

Brain defines a **closed, minimal, and non-extendable**
set of canonical epistemic classes.

This set represents the **irreducible epistemic surface** of the system.

No canonical system may:

- introduce new foundational classes
- remove existing canonical classes
- reinterpret class identity

All external models, ontologies, and domain semantics
**must map onto this set**, never extend it.

---

## 7. Closed-Set Rule

The canonical class set defined by Brain is **closed**.

Extensions are permitted **only as mappings**, never as additions.

Mapping rules:

- a mapping must declare source semantics
- a mapping must declare target canonical class
- a mapping must not alter canonical meaning

Failure to map implies non-canonical status.

---

## 8. Relationship to Core Authority

- **KSA** governs meaning of terms used within Brain objects
- **Regimes** govern admissibility and evaluation
- **Validity** operates over Brain objects
- **Interpretation** constrains how Brain objects are read

Brain does not override Authority.  
Authority does not redefine Brain.

They are orthogonal and mutually constraining.

---

## 9. Prohibition of Technical Implementation Concerns

At the Canon and Brain level:

- technical runtime descriptions are prohibited
- implementation models are prohibited
- storage schemas are prohibited
- execution flow definitions are prohibited
- tooling and infrastructure are prohibited

Any reference to **technical implementation-level concerns**
at this level constitutes a **canonical violation**.

---

## 10. Non-Equivalence Rules

- Brain ≠ Core
- Brain ≠ technical runtime
- Brain ≠ implementation
- Brain ≠ agent
- Brain ≠ intelligence
- Brain ≠ UI or projection

Brain is a **canonical derivative and runtime model**, nothing else.

---

## 11. Stability Clause

Brain is **stable by default**.

Changes to Brain:

- require explicit Canon amendment
- are versioned at Canon level
- are globally breaking by definition

Silent evolution of Brain is forbidden.
